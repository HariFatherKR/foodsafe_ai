비개발자용 안내서

Codex 설정, 딱 두 파일만 알면 됩니다

`AGENTS.md`는 "일하는 방식 설명서", `config.toml`은 "기본 동작 스위치"입니다. 이 페이지는 개발 용어를 최소화해서, 팀 운영 관점에서 바로 이해할 수 있게 정리했습니다.

최종 업데이트: 2026-02-16 | 기준: OpenAI Codex 공식 문서

1) `AGENTS.md`가 하는 일

"이 프로젝트에서는 어떤 순서로, 어떤 기준으로 작업할지"를 Codex에게 알려주는 안내문입니다. 팀 규칙이 바뀌면 여기부터 수정합니다.

2) `config.toml`이 하는 일

모델, 승인 방식, 샌드박스 같은 "동작 기본값"을 정하는 설정 파일입니다. 개인/프로젝트 공통 정책을 안정적으로 고정할 때 사용합니다.

어디에 두면 되나요?

  • 전역`~/.codex/AGENTS.md` : 모든 프로젝트 공통 작업 지침
  • 프로젝트`(프로젝트 루트)/AGENTS.md` : 해당 프로젝트 전용 지침
  • 전역 설정`~/.codex/config.toml` : 기본 실행 정책
  • 프로젝트 설정`(프로젝트)/.codex/config.toml` : 해당 프로젝트만 다른 정책

핵심 기억: 가까운 위치의 규칙이 보통 더 강하게 적용됩니다. (예: 프로젝트 폴더 안 설정이 전역 설정보다 우선되는 경우)

비개발자 기준 실무 운영 순서

  • Step 1전역 `AGENTS.md`에 조직 공통 원칙만 적습니다. (보안, 리뷰, 테스트 최소 기준)
  • Step 2각 저장소 `AGENTS.md`에 팀별 실행 방법을 적습니다. (빌드 명령, 배포 체크)
  • Step 3`config.toml`에 승인/샌드박스 정책을 고정합니다.
  • Step 4위험 명령은 `.rules`에서 `prompt` 또는 `forbidden`로 막습니다.

권장 운영: 공통 규칙은 짧게, 프로젝트별 예외는 로컬 파일에 분리.

바로 복붙 가능한 최소 템플릿

아래 두 블록만 먼저 적용해도 운영 품질이 빠르게 안정됩니다.

A. `AGENTS.md` (프로젝트 루트)

# AGENTS.md instructions for /absolute/path/to/repo

## Build & Test
- Install: `pnpm install`
- Test: `pnpm test`
- Lint: `pnpm lint`

## Safety
- Do not run destructive git commands without explicit approval.
- Never modify production secrets.

## Change Policy
- Any behavior change must include related docs and tests.

B. `~/.codex/config.toml` (전역)

model = "gpt-5.3-codex"
personality = "pragmatic"
model_reasoning_effort = "xhigh"
approval_policy = "on-request"
sandbox_mode = "workspace-write"

[projects."/absolute/path/to/repo"]
trust_level = "trusted"

운영 체크리스트 (회의용)

  • 전역 규칙과 프로젝트 규칙의 역할이 분리되어 있다.
  • 팀이 실제 쓰는 테스트/검증 명령이 `AGENTS.md`에 명시되어 있다.
  • 승인 정책(`approval_policy`)이 조직 보안 레벨과 맞다.
  • 위험 명령은 `.rules`로 최소 `prompt` 처리되어 있다.
  • 신규 프로젝트 시작 시 템플릿을 복사해 일관되게 적용한다.

자주 묻는 질문

Q1. 저는 개발자가 아닌데 꼭 `config.toml`까지 알아야 하나요?

네. 실제 사고는 코드보다 설정에서 나는 경우가 많습니다. 최소한 승인 정책과 샌드박스 모드 두 가지는 운영 담당자도 확인할 수 있어야 합니다.

Q2. 왜 파일을 전역/프로젝트로 나누나요?

공통 원칙(회사 정책)은 한 곳에서 유지하고, 프로젝트 차이(명령어/배포 방식)는 각 저장소에서 관리해야 변경 충돌이 줄어듭니다.

Q3. 문서가 너무 길어지면 어떻게 하나요?

전역 파일은 원칙만 남기고, 상세 절차는 프로젝트 파일로 내리세요. "누가 봐도 바로 실행 가능한 문장"만 남기면 유지보수가 쉬워집니다.
